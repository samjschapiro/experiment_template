# Per-experiment template

A single deployable rating study. The factory script (`../factory.py`) clones this directory once per condition / model / variant in a multi-arm campaign; for a one-off study just edit this directory directly.

## Files

```
README.md                  ← this file
index.html                 ← static page, loads jsPsych + experiment.js
js/
  experiment.js            ← timeline: consent → ratings × stimuli × dims → submit
  stimuli-data.example.js  ← reference for the bundle shape
  stimuli-data.js          ← (generated by prepare_stimuli.py; gitignored)
prepare_stimuli.py         ← upstream source → js/stimuli-data.js
vercel.json                ← frontend hosting config
deploy.sh aws|supabase     ← deploy backend, sed API_BASE into experiment.js, deploy frontend
get_data.sh aws|supabase   ← pull submissions to data/{experiment}.csv
filter_incomplete.py       ← drop participants who didn't finish
backend/
  README.md                ← swap guide
  aws/                     ← AWS SAM (S3 + Lambda + API Gateway)
  supabase/                ← Supabase (Edge Functions + Postgres)
```

## Quickstart (single experiment)

```bash
# 1. Define stimuli + rating dimensions
$EDITOR prepare_stimuli.py
python prepare_stimuli.py --input path/to/upstream.csv

# 2. Pick a backend, deploy
bash deploy.sh supabase   # or `bash deploy.sh aws`

# 3. Test in your browser
open https://<your-deployment>.vercel.app
# (no PROLIFIC_PID → debug mode; submissions disabled)

# 4. Wire to Prolific (or any recruiter)
# Study URL: https://<your-deployment>.vercel.app?PROLIFIC_PID={{%PROLIFIC_PID%}}&STUDY_ID={{%STUDY_ID%}}&SESSION_ID={{%SESSION_ID%}}

# 5. After recruiting
bash get_data.sh supabase
python filter_incomplete.py
```

## Editing the timeline

Open `js/experiment.js`. Sections to adjust:

| Section            | When to touch                                                         |
|--------------------|-----------------------------------------------------------------------|
| `CONFIG` block     | TOTAL_SLOTS / STIMULI_PER_SLOT, completion URL                        |
| `consent`          | Replace the consent text with your IRB-approved version               |
| `buildTimeline`    | Different stimulus rendering, different rating scales, attention checks |
| `getStimuliIndicesForSlot` | Different slot-balancing scheme (e.g. counterbalanced order) |

## What the backend stores

Every completed session → one record:

```json
{
    "participant_id": "65b4...",
    "slot": 7,
    "ratings": [
        {
            "stimulus_id": "s0001",
            "condition": "A",
            "rating_dimension": "dimension_1",
            "rating_value": 73.4,
            "rt": 4520
        },
        ...
    ],
    "client_metadata": {
        "user_agent": "...",
        "submitted_at": "2026-06-05T...",
        "debug_mode": false
    }
}
```

The `get-data` endpoint flattens this into row-per-rating CSV format. See `../analyze.py` for the z-scoring pipeline.
